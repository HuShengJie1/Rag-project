\begin{abstract}
随着工程教育专业认证（Engineering Education Accreditation, EAC）与成果导向教育（Outcome-Based Education, OBE）理念的深入推进，高校在开展专业建设与质量评估过程中产生了大量结构复杂的文档资料，如培养方案、课程教学大纲及达成度分析报告等。这类文档具有信息密集、结构复杂及证据链分散等特点，传统基于人工查阅的方式效率低下，且难以保证信息溯源的准确性。针对上述问题，本文设计并实现了一种基于检索增强生成（Retrieval-Augmented Generation, RAG）框架的工程教育专业认证私有知识助手系统。

在数据处理方面，本文采用语义感知切块（Semantic Chunking）方法对工程教育专业认证文档进行结构化处理，并为文本块绑定文件路径与页码等元数据，以支持证据链溯源。在检索机制方面，构建了“初筛—精排”的双阶段检索框架：首先基于 \texttt{bge-m3} 模型进行稠密向量检索以保证候选覆盖率，随后利用 \texttt{bge-reranker-v2-m3} 进行重排序，以提升候选文本的相关性。在生成阶段，设计了基于提示工程的约束机制，要求大模型仅基于检索内容进行回答并输出引用信息，从而降低幻觉风险并增强结果的可解释性。

在系统实现上，基于 Vue、LangChain 与向量数据库构建了完整的工程系统，并实现了证据链可视化功能。实验结果表明，所提出方法在检索排序质量（Mean Reciprocal Rank, MRR）及生成结果相关性方面均有所提升；同时，证据链相关指标显著提高，其中引用准确率与溯源成功率分别由 0.02 提升至 0.76 和 0.90，验证了系统在工程教育专业认证场景下的有效性与实用性。

综上，本文所提出的工程教育专业认证私有知识助手系统能够在保证检索效率的同时提升生成结果的可靠性与可追溯性，为复杂工程文档的智能化管理提供了一种可行方案。

\par\textbf{关键词：} 工程教育专业认证；成果导向教育；检索增强生成；双阶段检索；证据链溯源
\end{abstract}

\section{绪论}

\subsection{研究背景}
大规模语言模型（Large Language Model，LLM）自 Transformer 架构问世以来，凭借其在自然语言理解、上下文推理和文本生成等方面的卓越表现，已成为人工智能领域最具革命性的技术之一。LLM 在通用知识问答、摘要生成和代码辅助等知识密集型任务中的应用潜力获得了广泛认可。

然而，由于 LLM 的知识本质上是隐式存储于其数千亿级的参数之中，这种参数化记忆模式存在固有的挑战：首先，模型的知识库是静态且有时效性限制的，难以应对日新月异的实时信息；其次，LLM 在处理复杂或低频知识时，极易出现“幻觉”（Hallucination）现象，即生成看似合理但事实上错误或虚构的内容 \cite{gao_retrieval-augmented_2024}；更关键的是，其推理过程是一个“黑箱”，缺乏透明性和可溯源性，这对于需要严格证据链支撑的垂直应用场景是不可接受的 \cite{lewis_retrieval-augmented_2021}。

为了克服这些局限性，RAG 框架应运而生。RAG 代表了一种范式创新，它将 LLM 的内在（参数化）知识与庞大、动态的外部（非参数化）知识库进行协同融合。其核心机制是在 LLM 生成答案之前，通过智能检索器从外部数据库中提取最相关的证据或文档片段，并将这些证据作为上下文输入给 LLM 进行条件生成。RAG 框架通过这种“先检索，后生成”的方式，有效地解决了知识时效性、幻觉和透明度问题，从而保证了模型回答的高准确性、知识的实时性和证据链的可溯源性。这使得 RAG 成为将 LLM 安全、精准地引入到对可信赖性有极高要求的垂直领域的关键技术。

\subsection{研究意义}
本研究的特定背景为高校 EAC 工作，其核心理念是 OBE\cite{lzy__2014, lj__2015}。该工作具有知识密度高、文档类型复杂（如课程大纲、试卷、学生报告、自评报告等海量私有文档），以及对证据链完整性和溯源精准度有极高监管要求的特点。传统的认证工作流程高度依赖人工查阅，不仅耗费大量时间精力，且极易导致证据链遗漏或断裂，严重影响认证效率和质量。

基于此，设计并实现一个基于 RAG 框架的智能知识助手系统，以提供一个安全、精准、可追溯的工程教育专业认证知识服务平台，具有重要的现实意义：
\begin{itemize}
    \item \textbf{提升认证工作效率}：通过提供标准查询、证据链溯源等功能，能够帮助认证人员快速定位关联证据，显著缩短查阅时间，提升效率和准确度。
    \item \textbf{保障认证质量与促进持续改进}：系统能够确保证据链的完整且准确 \cite{lj__2015}，降低人为失误风险，同时支持对知识关联性的深度挖掘，辅助专业建设持续改进。
    \item \textbf{私有知识的安全与便捷管理}：利用向量数据库和 RAG 框架，可实现对海量分散私有文档的统一管理和高效检索，将文档转化为可被高效访问的知识资产。
\end{itemize}

\subsection{国内外研究现状}

\subsubsection{国外研究现状}
国外研究在大型语言模型的知识增强、向量数据库构建以及 RAG 性能优化方面处于理论和技术的前沿，为本领域提供了坚实的底层基础。

RAG 框架的提出是解决 LLM 知识局限性的关键突破。Lewis 等（2020）\cite{lewis_retrieval-augmented_2021} 开创性地提出了 RAG 模型，将预训练的 Seq2Seq 模型与可微分的外部非参数记忆相结合，证明了该模型能够显著提升 LLM 在知识密集型任务上的性能，同时解决了 LLM 决策不可溯源的问题，为模型的生成结果提供了证据来源（provenance）（见 \cite{lewis_retrieval-augmented_2021} Abstract）。在此基础上，Gao 等（2024）\cite{gao_retrieval-augmented_2024} 系统地将 RAG 的发展划分为 Naive RAG、Advanced RAG 和 Modular RAG 三个阶段（见 \cite{gao_retrieval-augmented_2024} Abstract），并强调 Advanced RAG 范式致力于对“检索、增强、生成”这三个核心环节进行全面、精细化的优化，是当前研究的主流方向。

支撑 RAG 框架核心检索功能的是向量数据库（Vector Database，VDB）。Taipalus（2024）\cite{taipalus_vector_2024} 和 Ma 等（2025）\cite{ma_comprehensive_2025} 的研究均指出，VDB 的出现是为了管理传统数据库难以处理的高维向量数据，并已成为现代 AI 系统中的关键组件，广泛应用于相似性搜索和智能问答系统（见 \cite{taipalus_vector_2024} Abstract, \cite{ma_comprehensive_2025} Abstract）。孙雨生和曾俊皓（2024）\cite{sys_rag} 进一步细化了 VDB 的关键技术，包括向量数据的嵌入生成、索引和检索（见 \cite{sys_rag} 结果/结论）。为了实现 VDB 的高性能检索，高效的近似最近邻搜索算法至关重要。其中，Jégou 等（2011）\cite{jegou_product_2011} 提出的乘积量化（Product Quantization, PQ）算法，通过将高维空间分解并进行量化，是实现大规模向量索引和检索，同时兼顾速度和内存效率的核心技术之一（见 \cite{jegou_product_2011} Abstract）。此外，针对工程教育专业认证文档这类复杂、长篇的文本，RAG 面临检索不准确和上下文碎片化的挑战，Luo 等（2025）\cite{luo_does_2025} 提出了 RetroLM 框架，创新性地引入了 KV-level 检索增强机制（见 \cite{luo_does_2025} Abstract），通过在 LLM 的键值缓存（Key-Value Cache, KV Cache）层面进行检索，显著提高了模型对检索结果的有效利用率，为本课题处理复杂的长文档关联性问题提供了技术参考。

\subsubsection{国内研究现状}
国内研究主要聚焦于 RAG 技术的实践应用和工程教育领域的核心理论探讨，为本课题的应用场景和优化方向提供了实践经验和理论基础。

在 RAG 应用实践方面，国内研究已将 RAG 技术成功应用于企业客服和垂直知识问答系统。例如，张丽静 等（2024）\cite{zlj_llmrag_2024} 在智能客服系统中的研究，验证了基于 LLM 和 RAG 实现智能客服的可行性，并强调系统在实际应用中必须具备可溯源性和安全性（见 \cite{zlj_llmrag_2024} 摘要）。同时，冯腾发（2025）\cite{ftf_rag_2025} 对 RAG 知识库在低资源领域的应用进行了深入研究，实践证明 RAG 系统的性能高度依赖于文档切块策略和检索增强技术的选择，这为本课题的系统调优提供了经验指导（见 \cite{ftf_rag_2025} 引言/正文）。

本课题的应用领域对知识的权威性、精准性和证据链的完整性要求极高。在工程教育领域的核心理念方面，李志义（2014）\cite{lzy__2014} 详细解析了工程教育专业认证的核心——OBE 理念，强调教育目标应以“学生毕业时应达到的能力和成果”为导向（见 \cite{lzy__2014} 第一段）。而 林健（2015）\cite{lj__2015} 则进一步指出，认证工作的关键挑战在于对证据链条的高效管理和准确溯源，即如何可靠地证明课程目标与毕业要求之间的关联是完整且有效的（见 \cite{lj__2015} 英文摘要）。针对工程教育专业认证文档条款间语义耦合性强、证据链条复杂的特点，孙如玉 等（2025）\cite{sry_hsr-ragrag_2025} 提出了 HSR-RAG（层级语义递归 RAG），该方法通过建立基于文档结构特征的多级语义解析模型和递归式语义聚合算法，有效解决了标准文本处理中存在的知识碎片化与语义关联弱化等局限性（见 \cite{sry_hsr-ragrag_2025} 摘要），为本课题设计 OBE 导向的结构化切块策略提供了关键的技术思路。

\subsection{研究主要内容与预期目标}
\subsubsection{研究主要内容}
(1) 基于结构化感知的复杂文档解析与知识库构建。 针对工程教育专业认证中涉及的海量多模态非结构化文档（如培养方案、自评报告、教学大纲等），本文将重点探讨基于文档逻辑结构的语义切块（Chunking）策略，以克服传统固定字符切分导致的语义破损问题。同时，通过设计对比实验，定量分析不同解析工具在处理复杂表格、跨页段落及多级标题时的提取准确度，最终构建一个支持结构化感知的高质量本地私有知识库。

(2) 面向工程教育专业认证领域的向量化检索与重排序机制设计。 为满足工程教育特定术语（如“毕业要求指标点”、“OBE”等）的精准语义匹配需求，本文将引入适合该垂直领域的 Embedding 模型，并构建专属测试集（Golden Dataset）对主流中英文 Embedding 模型进行交叉评估。在检索策略设计上，本文采用“初筛-精排”的双阶段架构：首先通过稠密向量（Dense Vector）检索召回初步相关的文档块；随后，引入基于 bge-reranker-v2-m3 的 Cross-Encoder 重排序（Re-ranking）机制，对初筛结果进行深度的语义交互计算与重新打分，从而最大化最终输入给大模型的上下文精准度，并有效滤除噪声干扰。

(3) 大模型接入与生成约束（提示工程）。 系统后端将基于 LangChain 框架接入高性能大语言模型（如 DeepSeek 应用程序编程接口（Application Programming Interface, API）），打通 RAG 核心链路。针对工程教育专业认证工作对严谨性的极高要求，本文专门设计了面向 OBE 理念的提示词（Prompt）模板，强制约束模型仅依据检索召回的上下文进行应答。旨在有效抑制大模型的“幻觉”现象，确保生成内容的真实性与逻辑一致性。

(4) 系统工程化部署与证据溯源实现。 在前端展示层面，采用 Vue 框架开发知识助手的 Web 交互界面，完成系统的整体工程化部署。该模块设计的核心聚焦于“可追溯性”，通过开发证据链可视化功能，确保系统生成的每一条回答均能提供精确的引用链接，支持用户一键跳转至原始文档的对应段落与页码，从而为认证专家的核查工作提供无缝的证据支撑。
\subsubsection{预期目标}
\begin{itemize}
    \item 完成原型系统开发： 成功搭建一套基于 Advanced RAG 架构的工程教育专业认证知识助手（Vue + DeepSeek API），支持自然语言流畅交互与私有文档的高效管理。

    \item 实现高精度知识检索： 通过双阶段检索与重排序优化，显著提升系统在处理“课程目标”、“达成度计算”等高频专业问题时的检索准确率（Recall与Precision指标达到预期基准）。

    \item 确保证据链精准溯源： 生成的问答结果必须具备高度的可解释性，实现精准的“证据闭环”，杜绝事实性编造。

    \item 提升认证业务效率： 验证系统在实际业务场景中的可用性，证明其相较于传统人工翻阅方式能大幅缩短信息比对与检索时间，有效赋能教学管理人员高质量完成认证材料的整编工作。
\end{itemize}

\section{相关理论与核心技术基础}
\subsection{大规模语言模型与提示工程}
\subsubsection{大语言模型（LLM）理论基础}
LLM 是构建于 Transformer 架构之上的深度神经网络模型，通过在海量文本语料上进行预训练，展现出卓越的自然语言理解（Natural Language Understanding，NLU）与生成（Natural Language Generation，NLG）能力
。LLM 的核心机制在于其集成的自注意力机制（Self-Attention），该机制赋予了模型捕捉长距离词项依赖关系的能力，使其能够深入理解复杂的语义结构并在生成文本时保持高度的上下文连贯性\cite{zhao_merino_2025}。

尽管 LLM 在通用任务中表现优异，但其知识存储依赖于预训练阶段形成的“参数化记忆”。
这种模式导致模型存在固有的局限性：一方面，模型无法实时更新预训练截止日期之后的知识，产生“知识滞后”现象；
另一方面，在处理专业性极强或长尾领域的知识密集型任务时，模型极易因内蕴知识匮乏而产生“幻觉（Hallucination）”，即生成看似逻辑合理但违背事实或虚假的内容\cite{zlj_llmrag_2024}。


\subsubsection{提示工程}
提示工程是指通过设计、优化和精炼输入给大模型的文本（Prompt），以引导模型展现出特定能力并约束其输出行为的技术范式
。在不改变模型权重参数的前提下，提示工程作为连接用户意图与模型生成能力的桥梁，决定了 LLM 在特定应用场景下的表现上限。有效的提示设计通常由指令（Instruction）、上下文背景（Context）和输出约束（Constraint）等要素构成，旨在将模型的通用推理能力锚定在特定的任务框架内。

\subsubsection{基于上下文的约束提示与幻觉抑制}
在知识问答系统中，缓解模型幻觉的关键技术路径在于基于上下文的约束提示（Context-Augmented Prompting）。
其理论核心是将模型的工作模式从“基于检索的生成”转变为“基于给定知识的阅读理解”，即通过将检索到的外部非参数化知识（如专业文档片段）注入到提示词中，为模型提供即时的参考依据\cite{gao_retrieval-augmented_2024}。

研究表明，通过以下特定提示策略可以显著抑制幻觉现象：
\begin{enumerate}
    \item \textbf{角色设定（Persona Prompting）：} 在提示词中明确设定模型作为专业领域专家（如工程教育专业认证专家）的身份，促使模型进入更严谨的推理模式。
    \item \textbf{上下文注入与指引：} 明确告知模型“以下是相关的参考文档”，并将检索到的文本块作为事实边界提供给模型。\cite{gao_retrieval-augmented_2024}。
    \item \textbf{严格负约束（Negative Constraints）：} 通过在指令中加入如“仅依据提供的上下文回答”、“如果信息不足，请回答不知道而非自行编造”等排他性约束，强制限制模型的发散性思维，从根本上缓解事实性错误的产生。
\end{enumerate}
这种基于上下文的约束机制能够有效地将 LLM 的推理范围限定在可验证的证据链内，从而大幅降低事实性错误的产生概率，确证输出内容的专业性与真实性。
\subsection{RAG 技术原理}




\subsubsection{RAG 的基本定义与核心价值}
RAG 是一种结合了预训练生成模型（参数化记忆）与外部知识库（非参数化记忆）的混合范式\cite{gao_retrieval-augmented_2024,lewis_retrieval-augmented_2021}。
LLM 虽然在自然语言理解上表现卓越，但由于其知识隐式存储于静态的权重参数中，面临着显著的局限性：一是“知识滞后”问题，LLM无法获取训练截止日期之后的实时信息；二是“幻觉（Hallucination）”现象，即模型在处理长尾知识或复杂事实时易生成虚假内容\cite{gao_retrieval-augmented_2024,lewis_retrieval-augmented_2021}。

研究表明，RAG通过在生成环节前引入检索机制，强制模型依据外部可信证据进行回答，从而有效抑制了事实性错误\cite{gao_retrieval-augmented_2024,lewis_retrieval-augmented_2021}。
这种“开卷考试”式的运作模式，不仅增强了回答的准确性，还为模型的决策过程提供了可追溯的证据链，实现了从“黑箱推理”向“透明论证”的转变\cite{gao_retrieval-augmented_2024,lewis_retrieval-augmented_2021}。

\subsubsection{RAG 的标准执行范式}
根据Lewis等人的研究，RAG的标准处理流程可概括为索引（Indexing）、检索（Retrieval）与生成（Generation）三个核心阶段：

\begin{enumerate}
    \item  \textbf{索引阶段（Indexing）：}这是知识库构建的基础。首先对多模态原始文档进行清洗与提取，将其转化为统一的纯文本格式；随后利用文本分割技术将长文档切分为语义完整的文本块（Chunks）；最后通过嵌入模型（Embedding Model）将文本块映射为高维向量，并存储于 VDB 中，以支持后续的相似度计算\cite{gao_retrieval-augmented_2024,ma_comprehensive_2025}。
    \item  \textbf{检索阶段（Retrieval）：}当接收到用户查询（Query）时，系统利用相同的嵌入模型将查询请求向量化。通过最大内积搜索（Maximum Inner Product Search，MIPS）或余弦相似度算法，从数据库中匹配出与该查询语义距离最近的Top-K个文本块.\cite{lewis_retrieval-augmented_2021}这一过程实质上是从海量数据中召回最相关的上下文信息。
    \item  \textbf{生成阶段（Generation）：}检索到的文本块与用户的原始查询被共同喂入提示词模板（Prompt Template），引导LLM在既定上下文约束下组织语言，生成最终答案\cite{gao_retrieval-augmented_2024,lewis_retrieval-augmented_2021}。在该阶段，参数化记忆与非参数化知识协同工作，确保输出内容既具备自然语言的流畅性，又具备事实的准确性。
\end{enumerate}
\subsubsection{技术演进：从 Naive RAG 到 Advanced RAG}
随着研究的深入，RAG技术已从最初的朴素模式（Naive RAG）演进至高级架构（Advanced RAG）\cite{gao_retrieval-augmented_2024}
。Naive RAG主要依赖简单的“检索-阅读”链路，但在实际应用中暴露出检索精度不足、召回内容噪声干扰严重以及LLM处理长上下文时易产生“中间丢失（Lost in the Middle）”等痛点\cite{luo_does_2025}。

为克服上述问题，Advanced RAG引入了更为复杂的优化机制。其核心演进逻辑在于从单一检索转向“初筛-精排”的双阶段检索架构：
\begin{itemize}
    \item \textbf{初筛（Recall）阶段：} 系统采用以稠密语义检索为核心的召回策略，并在候选召回后引入重排序机制以提升最终相关性。相较于依赖关键词匹配的稀疏检索方法，稠密向量检索更适合处理工程教育专业认证文档中大量同义表达、隐式语义关系与跨段落语义关联问题\cite{gao_retrieval-augmented_2024,lewis_retrieval-augmented_2021}。
    \item \textbf{重排序（Re-ranking）阶段：} 在初筛阶段召回大量文档块后，引入基于 Cross-Encoder 的重排序模型。重排序器会对召回的文档块与用户问题进行更深层的语义关联度打分并重新排序，滤除其中的不相关噪声数据。这一机制确保了输入 LLM 的上下文是最具针对性的证据，有效减轻了无关噪声对模型生成的误导\cite{gao_retrieval-augmented_2024}。
\end{itemize}
\subsection{向量数据库与检索技术}
\subsubsection{文本向量化表示（Embedding）原理}
在处理大规模非结构化数据时，文本向量化（Embedding）是实现机器理解语义的核心技术。其基本原理是通过深度学习模型将离散的文本符号映射至一个连续的高维向量空间中，生成高维稠密向量（Dense Vector）\cite{ma_comprehensive_2025}
。与传统的稀疏表示方法不同，稠密向量能够将语义信息压缩在有限的维度中（通常为 768 或 1024 维），从而有效地捕获词项间的上下文关联与深层语义特征\cite{ma_comprehensive_2025,taipalus_vector_2024}。

这种表示方法的优势在于它实现了“语义距离”向“空间距离”的转化。在向量空间中，语义相近的文本块由于具有相似的特征分布，其对应的向量在空间位置上会彼此接近\cite{taipalus_vector_2024,sys_rag}
。这种机制不仅解决了关键词匹配无法处理同义词、近义词的局限性，还为后续基于数学度量的知识检索奠定了表征基础。

\subsubsection{相似度计算机制}
在完成文本的向量化表示后，衡量两个向量在空间中的相关性成为检索的关键。在众多的距离度量指标中，余弦相似度（Cosine Similarity）被广泛应用于自然语言处理任务中\cite{taipalus_vector_2024}。
余弦相似度通过计算两个向量之间夹角的余弦值来衡量它们的方向一致性，其取值范围在 [-1, 1] 之间，值越接近 1 则代表两个向量的语义相关性越高，而忽略其绝对模长的差异。

除了余弦相似度外，常见的度量方式还包括欧式距离（L2 Distance）和内积（Inner Product）\cite{taipalus_vector_2024,ma_comprehensive_2025}。
对于归一化后的向量，这几种度量方式在排序结果上具有等价性。以余弦相似度为例，其数学计算公式如下所示：
\begin{equation}
    \text{similarity} = \cos(\theta) = \frac{\mathbf{A} \cdot \mathbf{B}}{\|\mathbf{A}\| \|\mathbf{B}\|} = \frac{\sum_{i=1}^{n} A_i B_i}{\sqrt{\sum_{i=1}^{n} A_i^2} \sqrt{\sum_{i=1}^{n} B_i^2}}
\end{equation}
通过上述公式，系统能够量化查询向量与知识库中文档向量的匹配程度，并按分值高低进行排序召回\cite{sys_rag}。

\subsubsection{向量数据库与近似最近邻搜索算法}
VDB 如 Milvus 和 Chroma，是专为存储、管理和高效检索大规模高维向量而设计的专用系统\cite{ma_comprehensive_2025}
。在小规模数据集下，可以使用精确的最邻近搜索（K-Nearest Neighbor, KNN）通过暴力扫描（Brute-force）计算所有距离。然而，由于向量检索面临“维度灾难”，当数据量达到百万级甚至亿级时，KNN 的时间复杂度 O(nD) 将导致检索延迟无法满足实时应用的需求\cite{jegou_product_2011}。

为了在检索速度与精度之间取得平衡，向量数据库普遍采用近似最近邻搜索（Approximate Nearest Neighbor, ANN）算法\cite{jegou_product_2011}
。ANN 通过预先构建索引结构（如分层可导航小世界图索引（Hierarchical Navigable Small World, HNSW）或倒排索引），将搜索范围限制在空间的一个子集中，从而实现毫秒级的响应。

其中，PQ 是一种极具代表性的有损压缩与索引算法\cite{jegou_product_2011}
。PQ 的核心思想是将高维空间分解为多个低维子空间的笛卡尔积，并对每个子空间分别进行聚类量化
。这种方法不仅大幅压缩了向量的存储空间，还允许系统在不解压原向量的情况下，通过查表操作快速估算近似距离，显著提升了大规模向量库的检索能效\cite{ma_comprehensive_2025}
。
\subsection{工程教育专业认证与 OBE 理念}
\subsubsection{工程教育专业认证的背景与目的}
EAC 是由非政府、非盈利的第三方组织对工程技术领域专业进行的正式认可，旨在评价其教育质量是否达到行业公认的最低合格标准\cite{lj__2015}
。随着我国正式加入《华盛顿协议》，具有国际实质等效性的专业认证已成为推动高等工程教育改革的关键抓手\cite{lzy__2014}
。认证的核心目的在于进一步提高工程教育质量，增强人才培养对经济社会发展的适应性，并最终实现工程教育的国际互认，提升毕业生的国际竞争力\cite{lj__2015}。

\subsubsection{OBE 的核心内涵}
工程教育专业认证遵循“成果导向（OBE）”、“以学生为中心”和“持续改进”三大核心理念，其中 OBE 理念被视为整个认证体系的灵魂
。OBE 指出，教学设计与实施的目标应聚焦于学生通过教育过程最终取得的学习成果（Learning Outcomes），而非单纯的知识积累。

在实践中，OBE 强调“反向设计”原则：由外部需求决定培养目标，由培养目标决定毕业要求，再由毕业要求决定课程体系
。其中，“毕业要求”是对学生毕业时应掌握的知识、能力和素质的具体描述，而“课程目标”则是支撑毕业要求的最小执行单元
。这种严密的映射关系要求课程体系中的每一门课程必须对实现特定的毕业要求有确定的贡献，从而确保全体学生在完成学业时均能达到预期的顶峰成果。

\subsubsection{业务痛点关联：证据链管理与溯源挑战}
尽管 OBE 理念提供了严谨的逻辑框架，但在实际的认证业务与自评报告撰写过程中，高校面临着严峻的挑战。认证体系要求专业必须通过“举证”的方式，严谨地回答每一项毕业要求是如何实现的，以及如何评价其实际效果
。这意味着认证工作不仅涉及教学大纲、培养方案等制度性文件，还包含海量的教学过程性文档，形成了一个庞杂且语义耦合极强的非结构化文档体系\cite{lzy__2014,lj__2015}。

在传统的文档管理模式下，由于各条款与证据点之间存在复杂的多对多映射关系，人工进行“毕业要求—课程模块—课程目标”的逻辑校验与证据核查效率极低，且难以确保证据链条的完整性与准确溯源。
当专家进行现场考查或审核自评报告时，对“证据链条的高效管理和准确溯源”提出了实时化、精准化的极高要求。
这种对海量文档语义理解与证据自动定位的刚性需求，暗示了传统人工治理方式已难以支撑 OBE 理念的深度落地，急需引入智能化手段实现知识的精准检索与可靠证成。
\section{知识助手系统需求分析与总体设计}
\subsection{系统需求分析}
EAC 的核心理念是 OBE，其认证过程具有知识密度高、文档类型复杂以及对证据链完整性和溯源精准度要求极高的行业特征。
传统的认证工作流程高度依赖人工查阅海量分散的私有文档，极易导致证据链遗漏或断裂，严重影响认证效率和质量。
为突破这一业务瓶颈，本系统以解决 EAC 认证工作中的实际痛点为导向，结合 LLM 与 RAG 技术，进行详尽的系统需求分析。

\subsubsection{功能性需求分析}
为构建安全、精准、可追溯的工程教育专业认证知识服务平台，有效辅助教学管理人员进行自评报告撰写和材料整编，本系统需满足以下核心功能性需求：
\begin{enumerate}
    \item \textbf{私有文档库管理与解析需求} 工程教育专业认证工作涉及的私有文档往往篇幅较长且类型复杂（如专业培养方案、教学大纲、课程目标达成度报告、学生试卷及自评报告等）。系统必须具备对这些海量分散的私有文档进行统一管理和高效转化的能力。
    \begin{itemize}
        \item \textbf{多模态复杂文档导入：} 系统需支持可移植文档格式（Portable Document Format, PDF）、Word 等多模态与多格式文档的批量导入与预处理清洗。
        \item \textbf{结构化解析与语义切分：}针对传统固定字符数切分（Fixed-size chunking）容易导致表格、跨页段落或“毕业要求指标点”等逻辑条目语义破损的问题，系统需支持基于文档层级特征的结构化解析与语义感知切片（Chunking），提取并保留文档的层级结构信息与完整语境。
        \item \textbf{向量化存储：}系统需支持将解析后的非结构化文本通过嵌入模型（Embedding）转化为高维向量，并存储于本地私有向量数据库（如Milvus或Chroma）中，将离散文档转化为可被高效访问的知识资产。
    \end{itemize}
    
    \item \textbf{智能问答与专业意图理解需求} 认证工作参与人员需要通过系统快速获取高度专业化的信息。系统不仅需要具备通用的自然语言处理能力，更需要针对工程教育领域的特定语境进行深度适配。
    \begin{itemize}
        \item \textbf{自然语言交互：}系统需提供友好的Web端交互界面，支持用户以自然语言的形式输入查询问题。
        \item \textbf{专业意图精准理解：}针对通用大模型在处理工程教育专业认证特定领域术语时匹配度不高（即长尾与噪声问题），系统需能精准识别并理解“毕业要求指标点”、“课程目标达成度”、“OBE”等特定专业术语的语义关联。
    \end{itemize}

    \item \textbf{证据链可视化与精准溯源需求} 工程教育专业认证工作对“证据链”的完整性与真实性具有极高的监管要求。由于大语言模型存在固有“幻觉”（Hallucination）缺陷，其黑箱推理过程对于需要严格证据支撑的垂直应用场景是不可接受的。因此，系统必须建立严密的证据倒查机制：
    \begin{itemize}
        \item \textbf{引文参考输出：}  系统在输出最终回答时，绝不能仅提供单一的总结性文本，必须强制同步输出其参考的引文片段与支撑依据。
        \item \textbf{特定段落与页码级溯源：} 必须赋予系统高度的“可解释性”与“可溯源性”。系统需支持溯源可视化功能，当认证专家或用户点击回答中的引文标注时，系统需能够直接跳转并精准定位至原始文档（如 PDF）的具体页码与特定段落。这一需求是辅助专家无缝进行证据核查、确保证据链闭环的核心支撑。
    \end{itemize}
\end{enumerate}

\subsubsection{非功能性需求分析}
在满足上述业务功能的基础上，系统还需在性能、安全性与可维护性方面达到以下非功能性指标：
\begin{enumerate}
    \item \textbf{准确性与安全性需求} 大语言模型在生成过程中存在固有的“幻觉”缺陷，其黑箱推理极易生成看似合理实则错误的内容，这对于需要严格证据支撑的工程教育专业认证场景是不可接受的。系统必须通过严格的提示工程（Prompt Engineering）进行约束，确保大模型的回答完全忠实于本地检索到的私有文档，彻底杜绝事实性编造。同时，系统需保障本地化部署架构的数据安全，防止高校核心教务数据的外泄。
    \item \textbf{系统性能与响应效率需求} 为保障知识助手的交互体验，系统整体响应延迟需控制在合理范围内。由于本系统在检索环节采用了复杂度较高的双阶段检索架构（即稠密向量初筛检索与重排序精排机制），且大模型的长文本生成需要一定算力开销，系统必须在检索高召回率与计算速度之间取得良好平衡，确保端到端的问答延迟能够满足用户的日常流畅使用。
    \item \textbf{系统的可扩展性与易用性需求}
        \begin{itemize}
            \item \textbf{易用性：} 系统需具备友好的 Web 端交互界面（基于 Vue 框架构建），保证页面布局直观、交互操作简便，降低非技术背景教务人员的学习门槛。
            \item \textbf{可扩展性：}考虑到各专业的培养标准与认证材料每年都会进行滚动更新，系统的底层架构设计必须方便后续私有知识库的动态更新、文档扩容及索引热重载，以长效支撑高校工程教育的持续改进机制。
        \end{itemize}
\end{enumerate}



\subsection{系统总体架构设计}
\begin{figure}[htbp]
    \centering
    \includegraphics[width=\linewidth]{figures/system_architecture.png}
    \caption{基于 RAG 的工程教育专业认证系统总体架构图}

    \begin{enumerate}
        \item \textbf{数据与存储层 (Data \& Storage Layer)：}
        
        本层是系统的底层数据底座与物理存储介质，主要负责多源异构数据的持久化管理。系统底层的数据来源主要涵盖 EAC 工作中的各类私有多模态复杂文档，如专业培养方案、教学大纲、课程目标达成度报告及学生试卷等（通常涉及 PDF、Word 等格式）。在核心存储组件设计上，系统采用“双轨并行”的混合存储架构：一方面，采用专门的高性能向量数据库（如 Milvus 或 Chroma）来持久化存储经过解析与语义切分后的文本块（Chunks）及其对应的高维稠密向量，从而将海量离散文档转化为可被高效访问的机器可读知识资产；另一方面，系统利用本地文件系统（Local File System）持久化归档原始多模态文档与结构化元数据（Metadata）。这种存储策略不仅支撑了高效的向量近邻搜索，更为后续前端的应用层实现“证据链精准溯源与页码定位”提供了坚实的物理文件支撑。

        \item \textbf{核心处理层 (Processing Layer)：} 
        
        本层作为 RAG 架构的引擎中枢，承担着离线私有知识库构建与在线智能检索的双重核心任务。在离线知识库构建阶段，针对工程教育专业认证文档长文本、多表格等结构复杂的特征，系统引入了基于文档层级特征的结构化解析与语义感知切块（Semantic Chunking）技术\cite{gao_retrieval-augmented_2024}。该方法有效克服了传统固定字符粗粒度切分导致的表格破损与跨页语义断裂问题，最大程度保留了“毕业要求指标点”等核心条款的完整上下文语境。在在线检索机制设计上，本系统全面拥抱 Advanced RAG 理念，构建了高精度的双阶段检索（Dual-stage Retrieval）架构：首阶段利用稠密向量搜索（Dense Retrieval）进行快速初筛，保障大规模私有库检索的广度与高召回率；次阶段引入 bge-reranker-v2-m3 模型作为 Cross-Encoder 进行深度重排序（Re-ranking）精排操作。重排序模型通过细粒度计算文档块与用户查询之间的深度相关性得分，有效滤除初筛阶段引入的无关噪声数据，确切保障最终输入给大模型的上下文证据具有极高的专业匹配度与准确率\cite{gao_retrieval-augmented_2024}。

        \item \textbf{大模型控制层 (Model \& Logic Layer)： }
        
        本层是整个知识助手的“认知大脑”与逻辑编排控制中心。系统引入了高度模块化的 LangChain 框架进行工作流编排（Workflow Orchestration），实现数据层、处理层模型与外部接口的串联与状态调度。在核心的生成逻辑控制上，本层通过定制化提示工程（Prompt Engineering），设计了契合 OBE 理念的严格系统级提示词（Prompt）模板。该模板强制要求大模型将重排序后的高精度本地上下文证据与用户的 Query 进行有机组装，并严格限制其发散性思维，要求生成内容仅来源于检索结果。最终，系统通过对接调用 DeepSeek API 执行可控的文本生成任务。这种生成约束框架不仅赋予了通用大模型处理工程教育特定领域逻辑的能力，更从根本上抑制了其固有的“幻觉”现象，确保了专业问答事实严谨、无编造内容。

        \item \textbf{应用与交互层 (Application Layer)：}
        
        本层是系统直接面向终端用户（如认证专家、高校教务管理人员）的前端展示与操作门户。系统前端基于渐进式 JavaScript 框架 Vue 构建 Web 交互界面，注重布局的直观性与非技术人员的易用性。应用层重点封装了两大核心业务模块：一是“多轮自然语言问答”模块，支持用户在维护历史对话上下文的情境下，以自然、流畅的口语化方式对 EAC 繁杂指标和课程目标等问题进行深度追问与探索交互；二是“证据链可视化溯源”模块（本系统的核心工程亮点），该模块承接底层向量元数据与文件存储映射，在渲染 DeepSeek 生成的最终答案时，同步输出带标号的引文参考标注。用户仅需点击对应的引文标签，Web 界面即可实现文档分屏联动，直接跳转并精准定位（包含引文高亮与具体页码跳转）到原始 PDF 文档的对应段落。此举赋予了模型回答高度的“可解释性”与“可溯源性”，完美契合工程教育专业认证业务中对材料严密互证与证据链闭环的极高监管要求。
        
    \end{enumerate}

\section{核心模块设计与对比实验分析}

\subsection{复杂结构文档解析与精细化切块策略}
EAC 文档具有高度非线性的排版特征，常包含培养目标与毕业要求支撑矩阵、课程目标与考核映射表、跨页制度条款以及标题、正文、表格、公式混排等复杂结构。若仍采用传统 RAG 系统中基于纯文本流的解析与固定长度切块方式，往往会破坏表格的行列对应关系、标题层级及上下文逻辑，进而在检索阶段造成关键证据缺失、约束对象混淆以及语义块边界失真等问题。基于此，本节围绕复杂结构文档的解析引擎选型与基于文档结构的精细化切块策略展开分析。

\subsubsection{解析引擎选型的外部基准参考与本地验证}

为了获得外部评测依据，本文首先参考多模态文档解析端到端评测基准 OmniDocBench 的评价框架。该基准面向真实 PDF 文档中的复杂版面场景，其总体评分（Overall）由文本编辑距离（TextEdit）、表格还原度（TableTEDS）与公式准确率（FormulaCDM）等核心指标构成，能够较好反映解析器在复杂结构恢复方面的综合能力。对于工程教育专业认证业务中占比极高的矩阵表格、制度条款与公式混排页面而言，表格树编辑距离相似度（Tree Edit Distance based Similarity, TEDS）及结构还原度（TEDS-S）尤其关键，因为它们直接关系到后续切块是否能够保持表内关系与章节边界的一致性。

\subsubsection{语义感知与精细化切块策略}

上述本地验证结果表明，复杂工程认证文档中的主要问题并不只是字符识别错误，更关键的是表格结构断裂、章节层级错位以及跨页内容分离。这意味着，即便解析结果在文本层面“可读”，若缺乏面向文档结构的切块机制，仍会在后续检索阶段造成语义不完整、约束关系丢失以及证据链断裂。因此，在获得高保真 Markdown 文本后，系统进一步摒弃传统的固定字符长度（Fixed-size Chunking）切分方式，转而采用基于文档树结构的“语义感知切块（Semantic Structure-aware Chunking）”策略，具体实现如下：

\begin{enumerate}
  \item \textbf{基于层级标签的物理切分：}  
  解析器会识别 Markdown 中的标题标记（如 \texttt{\#}、\texttt{\#\#}），系统将其作为硬切分边界（Hard Boundary），确保同一章节中的“课程目标”“毕业要求”“评价依据”等内容不会被机械性地从中间截断。

  \item \textbf{表格与正文的异构块保留：}  
  对于矩阵表格、课程目标与考核映射表、达成度统计表等结构化内容，系统优先保留其作为独立逻辑块的完整性，而不与相邻正文进行简单拼接。这样既可保持表内行列关系，又便于后续检索直接命中表格型证据。

  \item \textbf{元数据联级绑定（Metadata Injection）：}  
  为实现证据链的可视化溯源，系统在切块过程中提取当前区块的上下文属性，将对应的物理页码（Page Number）、所属章节树（Section Title）以及文件本地路径（Source Path）作为元数据与当前文本块强绑定。

  \item \textbf{约束性回退机制：}  
  仅当单一逻辑块的长度超过大模型上下文检索的最佳视窗（本文设定为 800 字符）时，系统才触发次级递归字符切分（Recursive Character Split），以避免过长块对检索精度造成干扰，同时保证语义边界尽量不被破坏。
\end{enumerate}

该策略通过“结构还原 + 语义绑定”的双重保障，使复杂工程认证文档在进入向量库之前便尽可能保持其逻辑完整性与证据可定位性，从而为后续的双阶段检索与高亮溯源奠定坚实的数据基础。


\subsection{领域知识向量化与 Embedding 模型选型}

在 RAG 框架中，文本向量化（Embedding）的质量直接决定了第一阶段稠密检索（Dense Retrieval）的召回上限。针对 EAC 场景中存在的长文本解析与特定专业术语匹配需求，本节提出一个两阶段的模型选型方法：首先从主流榜单中筛选出若干候选模型，再通过本地测试集验证选出最优模型。

\subsubsection{评测基准与选型约束条件}

为客观评估不同文本向量模型在中文检索任务中的表现，本文选取 C-MTEB（Chinese Massive Text Embedding Benchmark）作为统一评测基准。该基准在检索（Retrieval）维度包含 T2Retrieval、MMarcoRetrieval、DuRetrieval 等多个子任务，并统一采用归一化折损累计增益（Normalized Discounted Cumulative Gain, nDCG@10）作为评价指标，通过对各子任务进行等权平均得到综合检索能力得分\cite{c-mteb}。

结合本课题的业务实际，模型选型需严格满足以下工程约束：
\begin{enumerate}
    \item \textbf{数据隐私与本地化部署：} 高校工程教育专业认证材料（如试卷、达成度报告等）涉及内部教学数据，不可通过公网上传至第三方闭源 API（如 OpenAI Embedding），模型必须支持完全本地化私有部署。
    \item \textbf{开源属性：} 模型权重需完全开源，以便后续开展可能的领域数据微调（Fine-tuning）。
    \item \textbf{参数规模可控：} 考虑到常规服务器的显存限制，优先筛选参数量在 1B 以下，最高不超过 4B 的轻量级模型，以保障大规模文档索引构建的吞吐率。
\end{enumerate}

\subsubsection{榜单筛选与候选模型确定}

基于上述约束，本文从公开的中文 Embedding 榜单中筛选了近年来具有代表性的开源模型，并汇总了其在 C-MTEB 检索任务上的公开评测数据（表 \ref{tab:embedding_models}）。

\begin{itemize}
    \item \texttt{bge-large-zh-v1.5}：在 C-MTEB 检索分数上表现优异，参数量适中。  
    \item \texttt{bge-m3}：支持最长 8K tokens 的长文本建模，适合处理篇幅较长的认证文档。  
    \item \texttt{MiniCPM-Embedding}：分数较高，但计算开销较大，作为性能上界参考。
\end{itemize}

\begin{itemize}
    \item \textbf{综合性能均衡：} 相较于其他模型在不同指标上的偏置表现，bge-m3 在召回与排序能力之间保持较好的平衡，更适合作为通用候选召回模型。
    
    \item \textbf{长文本建模能力：} bge-m3 支持更长上下文输入，能够较好适应工程教育专业认证文档中跨段落与复杂结构的语义匹配需求。
    
    \item \textbf{模型生态一致性：} bge-m3 与第二阶段精排模型 \texttt{bge-reranker-v2-m3} 属于同一模型体系，在表示空间上具有一致性，有助于提升双阶段检索流程的整体协同效果。
    
    \item \textbf{工程部署可行性：} 在保证性能的同时，模型规模与计算开销适中，能够满足实际系统部署需求。
\end{itemize}


\subsection{混合检索与重排序（Re-ranking）机制}


工程教育专业认证场景下的知识查询任务具有显著的领域特征。一方面，用户问题通常围绕“课程目标—毕业要求支撑关系”“课程考核规范”“达成度评价依据”“持续改进周期”等制度性与关系性知识展开，其答案往往并非集中于单一连续段落，而是分散于培养方案、课程教学大纲、课程考核实施细则、毕业要求达成度分析办法及持续改进制度文件等多类文档中。另一方面，此类文档普遍具有篇幅较长、层级结构复杂、表格与正文交织、跨页关联显著以及制度条款表达相似度较高等特点，导致仅依赖单次向量相似度排序难以同时保证候选召回范围与最终证据精度。

基于上述特点，本文在检索模块中采用“初筛—精排”的双阶段检索机制。该机制首先利用稠密向量检索实现高效候选召回，再通过 Cross-Encoder 精排模型对候选集合进行细粒度相关性判别，从而在保证召回能力的基础上提升最终检索结果的证据有效性与可作答性。整体上，该设计既保留了稠密检索在大规模语义匹配中的效率优势，又增强了系统对复杂制度型查询的判别能力，为后续受约束生成模块提供更加可靠的证据输入。

\subsubsection{单阶段稠密检索的局限性分析}

在工程教育专业认证场景中，知识检索对象主要包括培养方案、课程教学大纲、毕业要求达成度分析报告、自评报告以及各类教学管理与质量保障制度文件。此类文档通常具有篇幅较长、结构层级复杂、表格与正文交织以及同一语义关系跨段落、跨页分布等特点。因此，用户查询往往并非针对单一文本片段，而是需要在多个候选证据之间完成语义匹配与关系判别。在此背景下，若仅采用单阶段检索机制，难以同时兼顾召回范围与证据精度。

本文系统在初筛阶段采用基于 Bi-Encoder 架构的稠密向量检索方法，以 bge-m3 模型对用户查询与文档切块分别进行独立编码，并在统一向量空间中执行相似度匹配。这类方法与 Dense Passage Retrieval（DPR）所代表的 dense retrieval 范式一致，其优势在于能够对文档表示进行离线预计算，并结合向量数据库实现大规模近似最近邻检索，从而具备良好的效率与扩展性\cite{karpukhin-etal-2020-dense}。同时，Sentence-BERT 的研究表明，Bi-Encoder 通过预先计算文本向量，可以显著降低大规模语义匹配任务的在线计算复杂度\cite{reimers-gurevych-2019-sentence}。

然而，Bi-Encoder 的高效率建立在“查询与文档独立表示”的机制之上，这也导致其在细粒度语义判别方面存在一定局限。首先，整体语义相似并不等同于能够构成有效证据支撑。在工程教育专业认证场景中，用户问题往往涉及明确的关系约束或规则限定，例如“某项命题限制适用于何种对象”“某类评价工作的执行周期是多少”“某制度条款是否直接回答当前问题”等。仅依赖向量相似度，模型虽然能够识别查询主题，却未必能够准确区分文本是否真正满足问题中的约束对象与语义焦点。相关研究也指出，在候选结果已经具有较高语义相关性的情况下，系统仍可能难以区分“真正正确的证据”与“表面相关但实际上错误的近似候选”，即 close false positives\cite{iyer-etal-2021-reconsider}。

其次，文档切块过程可能带来上下文割裂。尽管本文采用语义感知切块策略以尽量保持语义完整性，但在长文档与复杂结构场景中，关键证据仍可能分布于不同文本块甚至跨页位置。例如，在持续改进制度文件中，某一问题的完整回答可能同时依赖“毕业要求综合评价周期”与“课程体系局部调整周期”两个条款，这些内容往往位于不同条目、不同页甚至不同文档中。此时，单个文本块即使与查询语义接近，也可能仅覆盖证据链的一部分，难以独立支撑完整回答。复杂 PDF 文档解析研究同样表明，多类型页面、复杂布局、表格与公式混排等因素会进一步放大内容抽取与后续检索中的结构性偏差，从而加剧证据链分散与排序不稳的问题\cite{ouyang2024omnidocbenchbenchmarkingdiversepdf}。

再次，为保证召回覆盖率，初筛阶段通常设置较大的 Top-K 值。这一策略虽然有助于提高候选覆盖率，但也会引入更多“语义相关但不直接回答问题”的噪声片段。尤其在工程认证制度文本中，大量条款共享相似的表层词项，如“教材”“分值”“评价”“不得超过”“周期”等。当检索模型主要依据全局语义接近性进行排序时，容易将主题相近但任务无关的条款一并召回。这样虽然提高了候选覆盖率，却也增加了后续生成阶段接收无效上下文的风险。已有研究表明，Cross-Encoder 相较于 Bi-Encoder 具有更强的查询—文档联合判别能力；将 BERT 类模型用于 passage re-ranking 后，能够显著提升候选结果的排序质量\cite{Humeau2020Poly-encoders:,nogueira2020passagererankingbert}。

为避免局限性分析停留在抽象描述层面，本文结合端到端检索实验中的真实查询案例，对单阶段稠密检索在工程认证制度文本场景下的不足进行归纳，如表\ref{tab:dense_limit_cases}所示。所选案例均来自本文评测集中的真实测试问题，并基于单阶段检索（S1）与双阶段检索（S2）的实际返回结果整理得到。其中，案例 C1 体现了“语义相关但证据对象不一致”的问题，即单阶段检索虽然命中了与“重复率 30\%”相关的条款，但首位结果对应的是 A、B 两套试卷之间的重复率约束，而非“与前两届试题的重复率限制”；案例 C2 体现了高召回场景下噪声条款前移的问题，单阶段检索受到“教材”“分值”“最高不超过”等表层特征干扰，将教学绩效类条款排在真正命题规范之前；案例 C3 则反映了复杂制度文本中证据链分散的问题，问题同时涉及“毕业要求合理性评价周期”与“课程体系调整周期”，所需证据分别位于不同条款甚至不同页中，单阶段检索虽能召回相关片段，但排序前列被相近的“四年周期”条款占据，导致首位结果不具备直接作答性。

综合来看，单阶段稠密检索更适合作为候选召回模块，而难以单独满足工程教育专业认证场景对于“高召回—高精度—可溯源”同时兼顾的要求。因此，有必要在初筛结果基础上引入精排机制，以实现从“高召回”到“高精度”的检索优化。

\subsubsection{双阶段检索框架设计}

针对单阶段稠密检索在工程认证场景中存在的“语义相关但证据不足”“跨块跨页证据割裂”以及“高召回下噪声增多”等问题，本文在初筛结果基础上引入精排机制，构建“初筛—精排”双阶段检索框架。其中，初筛阶段采用 bge-m3 对查询与文档切块进行稠密向量编码，以保证候选召回的覆盖率；精排阶段采用 bge-reranker-v2-m3 对初筛得到的候选集合进行逐对相关性建模，从而提升最终排序结果的证据有效性与语义判别能力。

与仅依赖向量空间相似度的单阶段检索不同，精排模型在计算过程中能够同时接收查询与候选文本，并基于交互式编码方式建模二者之间的细粒度匹配关系。因此，该阶段不仅能够判断文本是否“主题相关”，还能够进一步区分其是否真正满足用户问题中的关系约束与回答需求。对于工程认证文档中常见的制度条款查询、命题规范查询、评价依据定位查询以及周期性规则归纳查询而言，这种联合判别能力具有明显优势。

本文双阶段检索机制的整体流程如图\ref{fig:two_stage_retrieval_framework}所示，可概括为以下四个步骤：

\begin{enumerate}[leftmargin=2em]
    \item \textbf{查询输入与向量化：}  
    用户在前端输入自然语言问题后，系统首先对查询文本进行标准化处理，并调用 bge-m3 编码器生成查询向量表示。

    \item \textbf{初筛候选召回：}  
    系统在向量数据库中对全部文本块执行相似度检索，返回 Top-K 候选片段。该阶段的目标是以较低代价覆盖尽可能多的潜在相关证据，为后续精排保留充分的候选空间。

    \item \textbf{交互式精排：}  
    对于初筛得到的候选集合，系统将每个“查询—候选文本块”对输入 bge-reranker-v2-m3 模型，计算更加细粒度的相关性分数。与初筛阶段不同，该过程基于查询与候选文本的联合建模，能够更有效地区分“主题相近”与“真正可作答”的证据片段。

    \item \textbf{结果输出与证据传递：}  
    系统根据精排得分对候选结果重新排序，并选取最终 Top-N 片段传递给生成模块。同时，候选片段所绑定的文件路径、页码等元数据也一并保留，以支持后续回答中的引用定位与前端证据链可视化展示。
\end{enumerate}

从系统功能角度看，双阶段检索的核心优势体现在两个方面。其一，初筛阶段负责“尽可能找全”，精排阶段负责“尽可能找准”，二者分工明确，既避免了仅使用 Cross-Encoder 直接对全库逐条匹配所带来的高昂计算成本，也克服了单阶段稠密检索在细粒度判别上的不足。其二，通过在精排后再将高质量候选片段输入生成模块，可以有效降低无关上下文对答案生成的干扰，增强系统回答的可信度与可溯源性。

在工程实现上，本文双阶段检索模块与文档处理模块、生成模块形成紧密耦合。文档在离线阶段经过语义感知切块后，不仅被编码为向量写入向量数据库，同时还绑定本地文件路径、页码等元数据；在线检索时，精排后的高质量片段连同这些元数据一起传递至后续模块，从而保证系统不仅能够回答“是什么”，还能够明确指出“依据来自哪里”。这一点对于工程教育专业认证场景尤为重要，因为该场景强调查询结果必须具备可验证性、可定位性和可追溯性。

\subsection{提示工程设计与生成约束}

在完成检索阶段后，系统并非直接将用户问题发送至大模型进行开放式回答，而是将用户查询、检索得到的候选文本块以及对应的文档元数据共同组织为生成输入。其中，候选文本块为模型提供回答依据，文档名、页码与本地路径等元数据则为后续引用标注与证据链溯源提供支撑。

然而，即使在引入 RAG 框架后，大语言模型在生成过程中仍可能出现“幻觉（Hallucination）”现象，即生成内容与检索证据不一致，或输出缺乏明确依据支撑的回答。已有研究表明，检索模块的引入并不必然消除模型幻觉，因此仍需在生成阶段进一步设计约束机制，以保证回答结果的可靠性、可解释性与可追溯性。

基于此，本文在生成阶段构建了由输入组织、提示约束、引用控制、边界限制与前后端协同组成的生成约束机制，其总体结构如图 \ref{fig:generation_constraint_framework} 所示。该机制以“用户查询 + 检索上下文 + 元数据”作为统一输入，通过系统级提示工程对模型回答范围、回答依据与输出格式进行显式约束，并结合引用标记、证据定位与高亮联动等机制，实现生成结果与源文档之间的可视化溯源。

具体而言，本文首先对输入内容的组织方式进行设计，以保证模型能够接收结构完整且可追溯的上下文信息；随后，通过约束性提示模板限制模型仅依据检索结果进行作答，并要求其在回答中保留引用信息；同时，当检索证据不足或无法支持用户问题时，系统将触发边界控制机制，避免模型生成缺乏依据的扩展性内容；最终，结合前端界面的引用定位、高亮显示与分屏联动功能，形成从“证据输入—受控生成—结果溯源”的完整闭环。

\subsubsection{基于检索约束的生成机制}

RAG 框架的核心思想是在生成阶段引入外部知识，使模型不再依赖参数化记忆，而是基于检索到的上下文进行回答。相关研究表明，引入可靠外部信息能够显著降低模型的幻觉率，并提升回答的事实一致性。

在本文系统中，生成模型的输入由以下三部分构成：

\begin{equation}
\text{Input} = \{\text{System Prompt}, \text{Query}, \text{Retrieved Context}\}
\end{equation}

其中，Retrieved Context 由双阶段检索模块输出的 Top-N 文本块组成，并附带对应的元数据（如文件路径与页码）。模型被明确约束只能基于该上下文进行回答。

\subsubsection{系统级 Prompt 约束设计}

为避免模型脱离检索内容进行自由生成，本文设计了如下系统级提示模板：

\begin{quote}
你是一个工程教育专业认证知识助手。请严格依据提供的参考材料回答问题。  
要求：  
（1）只能基于给定材料回答，不得使用外部知识；  
（2）若材料中未包含答案，请明确说明“未在材料中找到相关信息”；  
（3）回答中必须标注引用来源（页码或文档片段）。  
\end{quote}

该设计属于典型的“约束式提示（Constraint-based Prompting）”方法。研究表明，明确的指令约束与上下文限定可以显著降低模型生成不准确信息的概率。同时，通过限制模型输出来源，可以有效避免其基于概率分布进行“补全式猜测”。

\subsubsection{证据驱动的引用生成机制}

为了进一步增强回答的可解释性，本文在 Prompt 中引入“证据引用”要求，即模型在生成答案时必须标注对应的来源文本。具体实现方式为：

\begin{itemize}
    \item 在输入上下文中为每个文本块附加编号或元数据标识；
    \item 在 Prompt 中显式要求模型输出引用标记；
    \item 在前端系统中根据引用信息实现文档定位与高亮展示。
\end{itemize}

已有研究表明，在 RAG 系统中引入显式引用机制可以显著提升答案的可验证性，并帮助用户判断信息来源的可靠性。同时，这种“证据绑定生成”方式能够抑制模型生成无依据内容，从而降低幻觉风险。

\subsubsection{幻觉抑制机制分析}

综合来看，本文的生成约束策略主要从以下三个方面抑制幻觉：

\begin{itemize}
    \item \textbf{知识来源约束：} 通过限制模型仅使用检索内容进行生成，避免依赖参数化记忆；
    
    \item \textbf{指令约束：} 通过系统 Prompt 明确规定回答范围与行为规则，减少不确定性；
    
    \item \textbf{证据绑定：} 通过强制引用来源，使模型输出与具体文本片段建立对应关系。
\end{itemize}

相关研究指出，提示工程在幻觉控制中具有重要作用，合理设计 Prompt 可以显著降低错误生成概率。此外，RAG 框架通过引入外部知识源，本身也是当前缓解幻觉问题的重要技术路径之一。

\subsubsection{与系统证据链机制的协同作用}

在本文系统中，生成模块并非孤立存在，而是与前端“证据链可视化”模块形成闭环。具体而言，生成结果中的引用信息将直接映射至原始文档的具体位置，实现以下功能：

\begin{itemize}
    \item 支持页码级跳转与原文高亮；
    \item 提供可追溯的证据链路径；
    \item 提升系统输出的可信度与可验证性。
\end{itemize}

因此，提示工程不仅用于控制生成行为，还在系统层面与检索模块及前端展示模块协同作用，共同构建“检索—生成—溯源”的闭环机制。

综上，本文通过引入基于检索约束的提示工程策略，实现了对大模型生成行为的有效控制，在降低幻觉风险的同时，提升了答案的可解释性与证据链完整性。

\section{系统实现与整体性能评测}

\subsection{开发环境与系统部署}

本文所提出的工程教育专业认证私有知识助手系统基于 RAG 框架实现，整体采用前后端分离的工程化架构。系统在开发与部署过程中所使用的关键软硬件环境如下：

\begin{itemize}
    \item \textbf{开发语言与框架：} 后端采用 Python 语言开发，结合 LangChain 实现检索与生成流程编排；前端基于 Vue 框架构建交互界面；
    
    \item \textbf{向量数据库：} 采用 Chroma 作为向量存储引擎，用于存储文本块及其对应的稠密向量表示；
    
    \item \textbf{嵌入与重排序模型：} 第一阶段采用 \texttt{bge-m3} 进行向量化表示，第二阶段采用 \texttt{bge-reranker-v2-m3} 进行候选重排序；
    
    \item \textbf{大模型接口：} 系统调用 Kimi API 作为生成模型，并结合提示工程实现受约束回答；
    
    \item \textbf{运行环境：} 系统部署于本地服务器环境，支持图形处理器（Graphics Processing Unit, GPU）加速（如 NVIDIA RTX 系列），操作系统为 Windows 或 macOS。
\end{itemize}

为验证本文所提出方法的可实现性与工程适用性，本文完成了面向工程教育专业认证场景的系统部署与运行实现。系统整体部署结构如图 \ref{fig:deployment_architecture} 所示。前端负责文档上传、问题输入、结果展示及证据链联动；后端负责文档处理、检索调度与生成编排；底层依托向量数据库、本地文件系统与元数据共同支撑离线知识库构建和在线问答服务。

在离线阶段，系统首先接收用户上传的工程教育专业认证相关文档，包括培养方案、课程教学大纲、达成度分析报告等。随后，后端服务调用文档处理模块完成文档解析、语义感知切块与向量化处理，并利用嵌入模型生成文本块的稠密向量表示。处理后的文本块及其向量被写入向量数据库，以构建面向私有语料的知识索引；同时，原始文档被存储于本地文件系统，文件路径、页码、文档名等元数据被同步保存，为后续引用定位与证据链溯源提供基础支撑。

在在线阶段，用户通过 Web 前端提交问题后，后端服务调用检索与生成编排模块执行问答流程。系统首先在向量数据库中召回与问题相关的候选文本，再通过重排序模型对候选结果进行相关性重估，筛选出高质量上下文，并将其连同用户问题一并送入 Kimi API 完成答案生成。生成结果返回前端后，系统进一步结合元数据实现引用定位、页码映射与证据链联动展示，从而提升回答结果的可解释性与可追溯性。

通过上述部署方式，系统实现了从原始工程教育专业认证文档到结构化私有知识库的转换，并形成了“离线知识构建—在线检索增强问答—证据链可视化溯源”的完整运行闭环，为后续系统测试与性能评估提供了实现基础。

\subsection{测试数据集构建与分类说明}

为对本文所提出的工程教育专业认证私有知识助手系统进行较为系统的验证，本文构建了面向工程教育专业认证场景的黄金测试集。该测试集共包含 50 组问答对（Question-Answer Pairs, QA），问题与标准答案均由工程教育专业认证相关文档中的明确内容整理而成，并为每个问题同步标注对应的原文核心片段，用于后续回答正确性、引用准确性与证据链溯源能力的评估。

从数据来源看，本测试集覆盖了培养方案、教学管理制度、课程质量评价办法、课程目标达成度评价细则、学籍管理条例、毕业要求达成情况评价办法以及持续改进指导意见等多类工程教育专业认证相关材料。测试集共关联 14 份源文档，既包含培养方案等宏观培养文件，也包含课程评价、学籍管理与持续改进等制度性文件，能够较好反映工程教育专业认证场景下知识类型多样、规则约束明确、跨文档分散存储的特点。

\subsection{核心功能模块实现与界面展示}

本文系统围绕工程教育专业认证知识查询需求，设计并实现了多模块协同的功能体系，主要包括文档处理模块、检索模块、生成模块及前端展示模块。

\paragraph{（1）文档处理模块}

系统在数据预处理阶段采用语义感知切块（Semantic Chunking）方法，将长文档划分为语义相对完整的文本块。同时，为每个文本块绑定对应的元数据（包括文件路径与页码），以支持后续的证据定位与溯源。

\paragraph{（2）双阶段检索模块}

检索模块严格按照“初筛—精排”流程实现。首先利用 \texttt{bge-m3} 模型对查询与文本块进行向量化，并在向量数据库中执行 Top-K 检索；随后利用 \texttt{bge-reranker-v2-m3} 对候选结果进行逐对评分，筛选出高相关文本作为最终上下文。

\paragraph{（3）生成模块}

生成模块基于 Kimi 大模型接口实现。在输入端，系统将查询与检索上下文拼接，并结合预定义 Prompt 模板对模型进行约束，要求其仅基于检索内容生成回答，并输出对应引用信息，从而降低幻觉风险。

\paragraph{（4）前端展示模块}

前端采用 Vue 框架开发，实现了查询输入、结果展示及证据链可视化等功能。为提升界面的可用性与结果的可解释性，系统前端整体界面划分为左侧对话管理区、中间问答交互区和右侧原文定位区三个部分，分别承担历史会话组织、回答结果展示以及证据链溯源展示等功能。

其中，左侧区域主要用于展示历史对话记录与当前会话状态，支持用户对已有问题进行回溯查看，并在同一会话中持续发起新的查询请求；中间区域为核心问答交互区，主要负责展示用户输入问题及系统生成回答，并在回答中保留引用标记；右侧区域用于展示回答对应的源文档内容，当用户点击引用标记后，系统可自动跳转至原文对应页码并对相关文本进行高亮显示。前端界面各功能区域示意如图 \ref{fig:ui_left}、图 \ref{fig:ui_center} 和图 \ref{fig:ui_right} 所示。

为进一步验证本文所提出系统在工程教育专业认证场景下的实际应用效果，本文基于第 5.2 节构建的黄金测试集，对系统进行了案例化测试。测试内容覆盖术语定义、逻辑计算和政策流程三类典型问题，并重点考察系统在回答正确性、证据引用合理性以及证据链溯源能力方面的表现。具体测试过程中，研究者将测试问题输入系统，记录系统回答结果，并结合标准答案、原文证据片段及跳转定位情况对结果进行分析。

\subsubsection{术语定义类问题测试}

术语定义类问题主要用于验证系统对工程教育专业认证场景下基础概念与制度术语的识别、检索与准确回答能力。该类问题通常具有答案表述明确、原文依据清晰的特点，因此适合用于检验系统是否能够稳定完成“检索—生成—引用”这一基本问答流程。

案例 1 的测试问题为“课堂教学‘教学六环节’具体指什么？”，标准答案为“备课、讲授、讨论、答疑、作业、考核”，对应证据来源于《上海海洋大学本科教学督导工作条例》。系统返回的回答为“课堂教学‘教学六环节’具体指的是备课、讲授、讨论、答疑、作业、考核等六个课堂教学环节”，与标准答案在语义上保持一致，且能够完整覆盖原文中的六个组成部分。该结果表明，系统在处理定义型问题时，能够较准确地抽取源文档中的关键术语解释，并生成与原文一致的规范化表述。

案例 2 的测试问题为“毕业要求指标点（3）‘设计/开发解决方案’除了技术外还需考虑什么？”，标准答案指出，除技术实现外，还应考虑健康、安全与环境、全生命周期成本与净零碳要求、法律与伦理、社会与文化等因素。系统生成的回答较完整地复现了上述要点，并保留了多因素并列的表达结构，与培养方案中的原文内容基本一致。此类问题涉及毕业要求指标点的具体内涵，是工程教育专业认证场景下高频查询内容。测试结果说明，系统不仅能够检索到目标条款，还能够在生成阶段保持较好的信息完整性，避免将复杂定义简化为过于笼统的概括性回答。

\subsubsection{逻辑计算类问题测试}

逻辑计算类问题主要用于验证系统对规则性文本、数值关系及计算逻辑的理解能力。与术语定义类问题相比，该类问题不仅要求系统检索到正确条款，还要求其能够正确解释或复现原文中的数值关系与计算结果，因此更能体现系统对半结构化制度文本的处理能力。

案例 3 的测试问题为“课程目标达成情况计算时，主客观权重如何分配？”，标准答案为“客观评价占 0.8，主观评价占 0.2”，原文证据明确给出了计算公式。系统回答不仅正确指出了“客观评价占总评价的 80\%，主观评价占总评价的 20\%”，还进一步给出了对应的公式表达。该结果说明，系统能够从制度性文本中提取数值关系，并在回答中保留较完整的规则表达形式。这对于工程教育专业认证中大量涉及评价办法、达成度计算和规则条款的文档场景具有较强现实意义。

案例 4 的测试问题为“如果某门课程规模为 75 人且教学课时为 32，该老师能得多少教学任务分？”，标准答案为 38.4 分。系统首先检索到“60--90 人的每个教学课时计 1.2 分”的规定，再依据教学课时 32 完成乘法计算，最终给出“38.4 分”的结果。该案例表明，系统不仅具备规则检索能力，还能对简单数值关系进行较为正确的推导与表达。虽然此类计算并不复杂，但在实际教学管理场景下，相关问题往往分散于制度文件中，人工查找与换算成本较高，因此系统在此类问题上的表现具有一定的应用价值。

\subsubsection{政策流程类问题测试}

案例 5 的测试问题为“考试需要准备几套试卷？”，标准答案为“A、B 两套”。系统回答指出“考试应准备 A、B 两套试卷，其覆盖面、题型、题量和难度应基本一致，两套试卷中重复题目不得超过 30\%”。从结果看，系统不仅正确回答了核心问题，还进一步补充了原文中的配套要求，使回答具有更强的制度完整性。这说明系统在处理流程型问题时，不仅能够定位核心规定，还能保留与该规定密切相关的上下文信息，从而增强回答的可解释性。

\paragraph{（3）综合分析}

综合实验结果可以得到以下结论：

（1）重排序机制显著提升了检索排序质量（MRR 提升约 5.5\%），并间接提升生成答案的相关性；

（2）提示工程约束对生成内容的“证据绑定能力”具有决定性作用，是提升系统可解释性的关键因素；

（3）Dense-RAG 虽在 Faithfulness 指标上表现较高，但在证据链指标上表现较差，说明其生成结果缺乏可靠来源支撑；

（4）双阶段检索与生成约束的结合，使系统在“回答准确性”与“证据可追溯性”之间取得了更合理的平衡。

因此，在工程教育专业认证场景中，S3（完整系统）能够更好地满足实际应用对可靠性与可解释性的要求。

\section{总结与展望}
\subsection{全文工作总结}

本文围绕 EAC 与 OBE 背景下复杂文档知识管理与查询困难的问题，设计并实现了一个基于 RAG 框架的工程教育专业认证私有知识助手系统。

针对工程教育专业认证文档类型多样、结构复杂、证据链分散等特点，本文从文档解析、语义检索与生成约束三个层面展开研究，构建了完整的系统技术方案。主要工作总结如下：

（1）在数据处理层，针对工程教育专业认证文档结构复杂的问题，提出基于语义感知切块（Semantic Chunking）的文档处理方法，并为每个文本块绑定文件路径与页码等元数据，实现从原始文档到可检索知识单元的结构化转化。

（2）在检索机制方面，设计了“初筛—精排”的双阶段检索框架。首先基于 \texttt{bge-m3} 模型进行稠密向量检索，以保证候选文本的覆盖率；随后引入 \texttt{bge-reranker-v2-m3} 进行重排序，从而提升候选结果的相关性与排序质量。

（3）在生成阶段，针对大语言模型可能产生幻觉的问题，设计了基于提示工程的生成约束机制，通过限制模型仅基于检索内容回答并输出引用来源，有效提升生成结果的可靠性与可解释性。

（4）在系统实现层面，基于 Vue、LangChain 与向量数据库构建了完整的工程系统，并实现了证据链可视化功能，使用户能够从生成答案直接跳转至原始文档对应位置，从而验证信息来源。

（5）在实验评估方面，本文通过端到端实验验证了系统的有效性。实验结果表明，引入重排序机制能够提升检索排序质量，而提示工程约束显著增强了生成结果的证据一致性与可追溯性，从而使系统在工程教育专业认证场景中具备良好的应用价值。

综上，本文提出的工程教育专业认证私有知识助手系统在检索效率、生成质量与证据链可解释性之间取得了较好的平衡，为复杂工程文档的智能化管理与应用提供了一种可行方案。

\subsection{系统局限性与未来展望}

尽管本文所提出的系统在工程教育专业认证知识问答任务中取得了一定效果，但仍存在一些局限性，有待进一步改进。

（1）\textbf{文档解析能力仍存在提升空间。}  
当前系统主要基于通用解析工具进行文档结构提取，在处理复杂排版（如跨页表格、嵌套结构与扫描文档）时仍可能存在信息缺失或结构误识别的问题。未来可结合更先进的多模态文档解析模型，以提升复杂文档的结构化能力。

（2）\textbf{检索模型对领域语义的适应性有限。}  
虽然本文采用的 \texttt{bge-m3} 模型在通用语义检索任务中表现良好，但其未针对工程教育专业认证领域进行专门训练，在处理专业术语与隐式语义关系时仍存在一定局限。未来可通过领域数据微调或构建领域专用嵌入模型，进一步提升检索效果。

（3）\textbf{生成评测仍依赖有限指标。}  
本文主要采用 Faithfulness 与 Answer Relevance 等指标对生成结果进行评估，但这些指标仍难以全面反映复杂问答任务中的语义质量。未来可结合更细粒度的评测方法，如多维人工评估或领域专家评审，以获得更全面的性能评价。

（4）\textbf{证据链构建仍依赖文本粒度。}  
当前系统的证据链以文本块为基本单位，在处理跨块推理或多证据融合问题时存在一定局限。未来可探索基于图结构的知识表示方法，将文档内容组织为知识图谱，以支持更复杂的推理能力。

（5）\textbf{系统扩展性与实时性能有待优化。}  
随着知识库规模的不断扩大，系统在检索效率与响应时间方面可能面临挑战。未来可结合向量索引优化技术或分布式检索架构，以提升系统在大规模数据场景下的性能表现。

未来工作可从以下方向展开：

（1）引入多模态文档理解能力，实现对图表、公式及扫描文档的统一解析；  
（2）结合领域知识构建更精细的知识表示结构，提升复杂语义推理能力；  
（3）优化提示工程与生成策略，进一步降低幻觉风险；  
（4）提升系统在真实教学管理场景中的部署与应用能力。

综上，本文工作为工程教育专业认证文档智能化处理提供了初步探索，未来仍有较大提升空间，有望在更广泛的教育与知识管理场景中发挥作用。
