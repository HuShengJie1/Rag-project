# 截图页面最终说明

## 一、最适合直接截图进论文的 HTML 文件

当前最适合直接截图进论文的两个文件如下：

1. `experiments/4.1章/screenshots/parser_matrix_compare.html`
2. `experiments/4.1章/screenshots/parser_mixed_compare.html`

二者分别对应：

- 图 A：培养方案第 2 页“毕业要求支撑矩阵”对比；
- 图 B：制度办法第 1 页与第 4 页复杂结构页面对比。

如果当前更关注公式正常渲染，而不是并排卡片式展示，也可以直接使用以下 Markdown 版本：

1. `experiments/4.1章/screenshots/parser_matrix_compare.md`
2. `experiments/4.1章/screenshots/parser_mixed_compare.md`

## 二、每个 HTML 文件建议截图的区域

### 1. `parser_matrix_compare.html`

- 建议截图区域：
  - 页面标题；
  - 下方四列完整卡片区域。
- 建议用途：
  - 直接放入 4.1.1 本地解析对比部分，用于说明同一矩阵表格在不同解析方案下的结构差异。

### 2. `parser_mixed_compare.html`

- 建议截图区域：
  - 若版面允许，截取整页，保留两行对比；
  - 若版面有限，优先截取第 2 行“制度办法第 4 页，公式与表格混排页”。
- 建议用途：
  - 用于说明扫描型制度文档中，弱基线失效、pipeline 公式失真以及 MinerU2.5 结构保持更完整的差异。

## 三、为什么 HTML 形式比原始 Markdown 更适合论文插图

相较于原始 Markdown 摘录，本次生成的 HTML 页面更适合直接用于论文插图，原因主要包括：

1. 各方案采用固定宽度并排展示，版式统一，便于在同一张图中直接比较。
2. 对表格类结果直接渲染 HTML 表格，而不是退化为纯文本，从而能够更直观地展示结构保留差异。
3. 对纯文本结果使用等宽字体和保留换行的形式展示，既保留原始问题特征，又避免手工整理带来的失真。
4. 每个卡片底部都附有一句“小结”，截图后无需再额外在图注中重复解释全部细节。
5. 原始 PDF 页面也被放入统一版面中，读者可以在同一视图下看到“原页结构”与“解析输出”的对应关系。

如果需要让公式由 Markdown 预览器直接渲染，则 Markdown 版本更合适；如果需要一次性截出统一版式的对比图，则 HTML 版本仍然更方便。

## 四、当前是否仍缺少真实输出材料

当前用于截图的核心材料已经齐备，能够支撑“为什么最终选用 MinerU2.5”的直观展示。但仍有以下边界需要说明：

1. 制度办法第 1 页的 `pdftotext -layout` 输出真实为空。
   - 页面中已按真实情况标注为缺失。
2. 当前截图页面是围绕论文最需要展示的页面进行压缩编排，并未把所有已有方案全部堆叠进去。
   - 例如制度办法第 1 页未展示 pipeline，第 4 页未展示 `pdftotext -layout`，这是为了保证截图可读性。
3. 当前页面未额外生成裁剪后的独立局部图文件。
   - 原始 PDF 页面的“局部”效果通过页面图片与统一样式中的聚焦区域实现，打开页面即可截图。

## 五、结论

当前 `experiments/4.1章/screenshots/` 下的两个 HTML 页面已经能够直接服务于论文截图工作流，并且能较直观地支持最终结论：在本项目的复杂工程认证文档场景中，`MinerU2.5` 相比弱基线与传统 pipeline 更适合作为后续精细化切块与 RAG 检索的上游解析方案。
