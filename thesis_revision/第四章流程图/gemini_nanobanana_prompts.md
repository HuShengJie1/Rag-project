# Gemini Nano Banana 流程图提示词

本文档用于生成第 4 章“模块实现”部分的论文插图。所有提示词均严格基于当前项目真实实现整理，可直接复制给 Gemini Nano Banana 使用。

## 通用总提示词

先给模型这段总约束，再给后面的模块专用提示词：

```text
请为本科毕业论文绘制一张“黑白简洁学术风格”的中文流程图插图。

硬性要求：
1. 只画模块实现流程，不要画系统总架构图，不要画部署图。
2. 白底、黑线、灰度风格，避免彩色渐变、卡通风、3D 效果、网页 UI 风格。
3. 节点使用简洁矩形或圆角矩形，箭头清晰，整体纵向排布，适合论文页面插图。
4. 图中文字全部使用中文，单个节点文字尽量简短，不要一大段解释。
5. 节点数量控制在 6 到 9 个之间，主流程明确，不展开异常处理、日志打印和边缘情况。
6. 适合直接插入毕业论文第 4 章，风格要正式、工科、克制，不要花哨装饰。
7. 需要体现输入、核心处理步骤、中间数据形态、输出；如果有关键元数据，也要在图中简洁标出。
8. 不要引入项目中不存在的技术，不要出现 BM25、CoT、知识图谱、多智能体之类未使用内容。
9. 项目真实技术路线是：MinerU2.5、bge-m3、bge-reranker-v2-m3、Kimi API、LangChain、Vue、Chroma。
10. 输出一张干净、平衡、适合论文截图的流程图，不要给说明文字，不要多版本，只给最终图。

版式要求：
- 纵向主流程
- 节点居中对齐
- 箭头从上到下
- 中间数据节点可用虚线框或稍浅边框区分
- 留白充分，避免过满
- 图像宽高比例适合 A4 论文页面
```

## 1. 文档解析与语义切块模块

```text
请根据以下真实系统流程，绘制“文档解析与语义切块模块实现流程图”。

流程必须严格基于以下实现：
1. 输入文件：PDF、DOCX、MD、TXT
2. 根据文件后缀判断处理路径
3. PDF 和 DOCX 调用 MinerU2.5 解析为 Markdown
4. MD 和 TXT 直接读取文本
5. 保存完整 Markdown，供后续证据回溯和高亮使用
6. 基于 Markdown 标题层级进行一级语义切块
7. 对超长块执行递归切分，控制检索粒度
8. 绑定元数据：file_id、user_id、headers、page_labels、source、full_md_path
9. 输出 Chunk 列表，形式是 text + metadata

画图要求：
- 标题感要像“论文中的模块实现流程图”
- 重点突出：MinerU2.5、Markdown、语义切块、元数据绑定
- 可以在一个中间数据节点中写“完整 Markdown”
- 可以在最后一个输出节点中写“Chunk 列表（text + metadata）”
- 关键元数据要简洁展示：headers、page_labels、source
- 不要出现 OCR、版面分析 benchmark、模型评测等无关信息
- 不要画得太复杂，保持 7 到 8 个节点
```

## 2. 领域知识向量化模块

```text
请根据以下真实系统流程，绘制“领域知识向量化模块实现流程图”。

流程必须严格基于以下实现：
1. 输入数据为上一步输出的 Chunk 列表及其元数据
2. 加载本地 bge-m3 嵌入模型
3. 初始化 Chroma 向量数据库
4. 整理待写入数据，形成 texts 和 metadatas
5. 调用 Chroma add_texts 执行向量化写入
6. 写入的关键元数据包括：chunk_id、source、page_labels、headers、full_md_path
7. 输出结果为可检索的向量索引

画图要求：
- 明确体现当前项目实际向量数据库是 Chroma，不要画 Milvus
- 明确体现嵌入模型是 bge-m3
- 中间可以放一个“向量与元数据”数据节点
- 输出节点写“可检索向量索引”
- 风格保持黑白简洁，适合本科论文
- 节点数量控制在 6 到 7 个
- 不要引入训练、微调、蒸馏、离线评测等流程
```

## 3. 双阶段检索与重排序模块

```text
请根据以下真实系统流程，绘制“双阶段检索与重排序模块实现流程图”。

流程必须严格基于以下实现：
1. 输入问题：prompt + source_filter
2. 根据 user_id 和 file_id 构造过滤条件
3. 在 Chroma 中执行相似度初筛检索，返回 Top-30 候选文本块
4. 形成候选集合，数据形态为 page_content + metadata
5. 构造 Query-Chunk 对
6. 调用 bge-reranker-v2-m3 进行重排序
7. 选取最终 Top-K 高相关证据
8. 组装 evidence_list，包含 index、source、pages、full_content、score
9. 输出到生成模块

画图要求：
- 图中明确体现“双阶段”：初筛检索 + 重排序
- 初筛阶段写 Chroma，相似度检索；精排阶段写 bge-reranker-v2-m3
- 中间数据节点可以写“候选集合”
- 输出节点可以写“evidence_list / 最终证据”
- 关键元数据简洁标出：source、pages、score
- 不要引入 BM25、关键词检索、混合稀疏检索等不存在内容
- 维持 7 到 8 个节点，纵向清晰排布
```

## 4. 受约束生成与证据引用联动模块

```text
请根据以下真实系统流程，绘制“受约束生成与证据引用联动模块实现流程图”。

流程必须严格基于以下实现：
1. 输入数据：用户问题、历史记录、evidence_list
2. 读取完整 Markdown 内容，补齐 full_content
3. 构造系统 Prompt，要求模型只能依据参考资料回答
4. 拼接编号上下文，把 [n]、来源、页码、正文内容组织成参考资料
5. 通过 LangChain 调用 Kimi API 执行流式生成
6. 先返回 evidence 元数据，再返回回答文本
7. 前端 Vue 将回答中的 [1][2] 解析为可点击引用标记
8. 根据编号映射回 source、pages、full_content
9. 展示原文并执行高亮联动

画图要求：
- 图中明确体现：Prompt 约束、Kimi API、引用编号、前端联动
- 这是“生成 + 引用联动”流程，不只是大模型生成
- 可以设置一个中间数据节点：流式返回结果（evidence + answer）
- 最终输出节点写“带引用回答与原文高亮联动”
- 关键元数据简洁标出：index、source、pages、full_content
- 不要画成聊天界面 UI，不要画成人机对话插画
- 不要引入不存在的后处理模块或规则引擎
- 节点保持 7 到 8 个，正式、清晰、可用于论文
```
