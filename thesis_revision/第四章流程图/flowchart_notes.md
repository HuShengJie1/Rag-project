## 1. 文档解析与语义切块模块实现流程图

### 节点含义

- `输入文件`：上传的 PDF、DOCX、MD、TXT 文件。
- `按文件后缀选择解析路径`：依据文件后缀判断是否调用 MinerU2.5。
- `完整 Markdown 文本`：得到解析后的统一文本形态，并持久化保存。
- `按标题层级进行一级语义切块`：以 Markdown 标题为边界组织文本。
- `对超长文本块执行递归切分`：控制单块长度，便于后续检索。
- `绑定关键元数据`：写入 `headers`、`page_labels`、`source`、`full_md_path` 等字段。
- `输出 Chunk 列表`：形成后续向量化输入。

### 为什么这样组织

图中先画“统一 Markdown 化”，再画“结构化切块”，最后画“元数据绑定”，符合项目真实入库顺序，也最能体现该模块对后续检索与证据链的支撑作用。

### 最重要的节点

- `按文件后缀选择解析路径`
- `按标题层级进行一级语义切块`
- `绑定关键元数据`

## 2. 领域知识向量化模块实现流程图

### 节点含义

- `输入数据`：接收上游 Chunk 列表。
- `加载 bge-m3 嵌入模型`：初始化嵌入模型和 LangChain 接口。
- `初始化 Chroma 向量库`：连接本地向量存储。
- `整理写入对象`：准备 `texts` 与 `metadatas`。
- `调用 Chroma add_texts`：执行入库。
- `向量与元数据`：形成可持久化的向量记录。
- `可检索向量索引`：作为在线问答检索底座。

### 为什么这样组织

该图强调的是“嵌入模型加载 + Chroma 入库”一体化流程，而不是将向量计算和数据库写入人为割裂。这与当前项目真实实现保持一致。

### 最重要的节点

- `加载 bge-m3 嵌入模型`
- `调用 Chroma add_texts`
- `可检索向量索引`

## 3. 双阶段检索与重排序模块实现流程图

### 节点含义

- `输入问题`：接收问题文本和来源筛选条件。
- `构造过滤条件`：根据 `user_id` 和 `file_id` 约束检索范围。
- `Chroma 相似度初筛检索`：进行第一阶段候选召回。
- `候选集合`：形成含正文与元数据的候选块。
- `构造 Query--Chunk 对`：为重排序准备输入。
- `调用 bge-reranker-v2-m3`：进行第二阶段精排。
- `evidence_list`：产出编号化证据包。
- `最终 Top-K 证据`：传给生成模块。

### 为什么这样组织

该图采用“过滤 -> 初筛 -> 精排 -> 证据组装”的顺序，既符合后端代码执行顺序，也便于论文中突出双阶段检索的工程实现逻辑。

### 最重要的节点

- `Chroma 相似度初筛检索`
- `调用 bge-reranker-v2-m3`
- `evidence_list`

## 4. 受约束生成与证据引用联动模块实现流程图

### 节点含义

- `输入数据`：问题、历史记录和证据列表。
- `读取完整 Markdown 内容`：补齐 `full_content` 供前端高亮使用。
- `构造系统 Prompt`：限制模型仅基于证据作答。
- `拼接编号上下文`：把编号、来源、页码、正文拼成参考资料。
- `LangChain 调用 Kimi API`：执行流式生成。
- `流式返回结果`：先发证据元数据，再发回答文本。
- `Vue 解析引用标记并点击映射`：将 `[n]` 转成前端可点击标记，并回查对应证据。
- `带引用回答与原文高亮联动`：输出给用户的最终效果。

### 为什么这样组织

该图不是单纯画“模型生成”，而是把后端 Prompt 约束、证据编号和前端回查联动放在同一链路中，才能体现本系统证据链闭环的真实实现。

### 最重要的节点

- `构造系统 Prompt`
- `拼接编号上下文`
- `Vue 解析引用标记并点击映射`
